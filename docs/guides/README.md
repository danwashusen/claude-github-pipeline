# Authoring guides for github-pipeline project docs

These guides teach a **consuming repo's** maintainer how to write the project documents the
**github-pipeline** skills read. The shared premise: your primary reader is an LLM — chiefly the
`github-issue-planner` — and the `github-issue-resolver` builds from the plan the planner emits,
not from your docs. So a fact your docs fail to make *citable* becomes a gap in the plan. Write
for citation, not narration.

The guides are **stack-neutral**; every worked example uses **Ruby on Rails 8.1** and is clearly
labeled as illustrative. Each follows the same spine: *why this doc exists → how the pipeline
consumes it → authoring principles → what belongs here vs. siblings → recommended outline →
worked example → checklist → anti-patterns.*

## The guides

| Doc (in your repo) | Guide | What it is | How the pipeline treats it |
|---|---|---|---|
| `docs/prd.md` | [prd.md](prd.md) | Product truth — *what* the product does and *why* | **Upstream anchor**: the drafter grounds every issue in it; re-checked downstream. Authoritative-but-mutable (conflicts route to a gate) |
| `docs/architecture.md` | [architecture.md](architecture.md) | *How* to build — layers, stack decisions, APIs | **Deviable default**: planner cites it; a departure needs a user-approved deviation |
| `docs/architecture-notes.md` | [architecture-notes.md](architecture-notes.md) | The *why* behind the architecture — rationale, rejected alternatives | Read **on-demand** when the planner is tempted to deviate; rationale, not rules |
| `docs/ui-design.md` | [ui-design.md](ui-design.md) | Authority for UI surfaces — named components, sizes, patterns | Deviable default; planner cites `ui-design §X` so the build reuses named components instead of inventing them |
| `docs/constitution.md` | [constitution.md](constitution.md) | Inviolable engineering rules | **Hard blocker**: a violation is rejected pipeline-wide, never negotiated. Typically `@`-included by `CLAUDE.md`, so keep it **very concise** |

## The two distinctions that decide where content goes

Most "which doc does this belong in?" questions resolve on two axes:

1. **Deviability.** Would you ever approve a deviation? *Never* → `constitution.md` (hard
   blocker). *Occasionally, with approval* → `architecture.md` / `ui-design.md` (deviable
   default). *It's product behavior that could change* → `prd.md` (authoritative-but-mutable).
2. **What / why / how.** *What the product does* → `prd.md`. *How it's built* →
   `architecture.md`. *Why it's built that way* → `architecture-notes.md`. *What it looks like* →
   `ui-design.md`.

A build/test/lint **command** belongs in none of these — it goes in a `CLAUDE.md` / `COMMANDS.md`
marker block (generated by `github-pipeline-setup`), which the resolver and evaluator read at
run-time. The docs state the *rule* or *requirement*; the marker blocks hold the *command*.

## Principles common to all five

- **Stable, numbered section anchors.** The planner cites `architecture.md §3`, `PRD §4`,
  `constitution §5`; renumbering dangles citations already posted in issues, plans, and reviews.
- **One governing claim per section**, stated prescriptively — so a citation resolves to a single
  checkable rule and a reviewer can verify the work against it.
- **No open questions or TBDs** — an ambiguity becomes a planner hedge that forces a round-trip.
- **Pin versions and relied-on defaults** where they matter, so the planner grounds on stated
  truth instead of guessing.

The [architecture.md guide](architecture.md) carries the fullest treatment of these; the others
adapt them to their doc's role.

## Where these live & how to validate

Put each doc at its `docs/*.md` path in the consuming repo. `CLAUDE.md` typically `@`-includes
`docs/constitution.md` (which is why it must stay short). To validate a doc against its real
consumer, run the skill that reads it on a representative issue and inspect the result — each
guide's checklist ends with the specific success signal to look for.
